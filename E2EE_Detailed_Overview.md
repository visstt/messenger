# End-to-End Encryption (E2EE) в этом мессенджере

## Что это значит

Сквозное шифрование в этой реализации означает:

- сообщение шифруется на устройстве отправителя;
- сервер получает и хранит только зашифрованные данные;
- расшифровка происходит только на устройстве участника чата;
- сервер знает только метаданные, необходимые для доставки и синхронизации.

Идея простая: бэкенд нужен для маршрутизации, хранения ciphertext и realtime-событий, но не для чтения содержимого переписки.

---

## Какие ключи используются

В системе используются 2 уровня ключей:

1. `RSA-OAEP 2048 / SHA-256`
   Используется для шифрования симметричного ключа отдельно для каждого участника чата.

2. `AES-GCM 256`
   Используется для шифрования содержимого конкретного сообщения или набора вложений.

Почему так:

- RSA удобно использовать для передачи ключа нескольким участникам;
- AES-GCM быстрее и подходит для реального шифрования текста и бинарных данных;
- у каждого сообщения свой отдельный AES-ключ.

---

## Где хранятся ключи

### На клиенте

При первом входе пользователя фронтенд:

1. генерирует пару ключей RSA;
2. сохраняет `privateKey` и `publicKey` в `localStorage`;
3. отправляет на сервер только `publicKey`.

Локальное хранилище:

```text
localStorage["messenger-e2ee-keypair-v1:<userId>"]
```

Пример структуры:

```json
{
  "publicKey": "{\"kty\":\"RSA\",\"e\":\"AQAB\",\"n\":\"...\",\"alg\":\"RSA-OAEP-256\",\"ext\":true}",
  "privateKey": "{\"kty\":\"RSA\",\"d\":\"...\",\"p\":\"...\",\"q\":\"...\",\"dp\":\"...\",\"dq\":\"...\",\"qi\":\"...\",\"alg\":\"RSA-OAEP-256\",\"ext\":true}"
}
```

### На сервере

Сервер хранит только публичный ключ пользователя:

```text
users.public_key
```

Это нужно, чтобы отправитель мог зашифровать симметричный ключ для всех участников чата.

---

## Что именно шифруется

### Текстовые сообщения

Шифруется весь полезный payload сообщения:

```json
{
  "text": "Привет! Встречаемся в 19:30",
  "attachments": []
}
```

### Вложения

Для вложений шифруются:

- подпись к вложению;
- имена файлов;
- MIME-типы;
- размеры;
- IV каждого вложения;
- длительность голосового сообщения;
- само бинарное содержимое файла.

Пример payload для изображения:

```json
{
  "text": "Вот скрин",
  "durationSec": 0,
  "attachments": [
    {
      "name": "screen.png",
      "mimeType": "image/png",
      "size": 182044,
      "iv": "vP9lM4cJ0mE4F9uY"
    }
  ]
}
```

Содержимое самого файла шифруется отдельно тем же AES-ключом, но со своим `iv` из `attachments[]`.

---

## Что не шифруется

Чтобы система могла работать, часть данных остаётся доступной серверу:

- `chatId`;
- `senderId`;
- `messageId`;
- `kind` (`text`, `image`, `file`, `voice`, `video`);
- время создания;
- статус удаления/редактирования;
- факт существования сообщения;
- URL загруженного зашифрованного файла;
- состав участников чата;
- публичные ключи участников.

То есть сервер не видит **содержимое**, но видит **структуру общения и служебные метаданные**.

## Поток отправки текстового сообщения

### 1. У отправителя уже есть публичные ключи участников

Например, в чате 2 участника:

```json
[
  { "id": 1, "name": "Alice", "publicKey": "{...Alice public JWK...}" },
  { "id": 2, "name": "Bob", "publicKey": "{...Bob public JWK...}" }
]
```

### 2. Клиент формирует открытый payload

```json
{
  "text": "Код готов, проверь пожалуйста",
  "attachments": []
}
```

### 3. Генерируется одноразовый AES-ключ сообщения

Схематично:

```text
messageAesKey = random(256-bit)
payloadIv = random(12 bytes)
```

### 4. Payload шифруется через AES-GCM

Результат:

```json
{
  "encryptedPayload": "yW8J7lM4Yx8G...base64...",
  "iv": "s6Q9YHfN2kW9dE3B"
}
```

### 5. AES-ключ отдельно шифруется для каждого участника через их public key

Смысл:

- Alice сможет расшифровать свой экземпляр AES-ключа;
- Bob сможет расшифровать свой экземпляр AES-ключа;
- сервер не сможет расшифровать ни один.

Пример:

```json
{
  "keys": {
    "1": "QWxpY2UtZW5jcnlwdGVkLUFFUy1rZXk=",
    "2": "Qm9iLWVuY3J5cHRlZC1BRVMta2V5"
  }
}
```

### 6. На сервер уходит только ciphertext

Пример запроса:

```json
{
  "text": "",
  "encryptedPayload": "yW8J7lM4Yx8G...base64...",
  "encryptionMeta": "{\"version\":1,\"algorithm\":\"AES-GCM+RSA-OAEP-256\",\"iv\":\"s6Q9YHfN2kW9dE3B\",\"keys\":{\"1\":\"QWxpY2Ut...\",\"2\":\"Qm9iLWVu...\"}}"
}
```

Обрати внимание:

- поле `text` пустое;
- настоящий текст лежит только внутри `encryptedPayload`.

## Поток отправки файла

Рассмотрим отправку файла `contract.pdf`.

### Открытые данные до шифрования

```json
{
  "text": "Подписанный договор",
  "durationSec": 0,
  "attachments": [
    {
      "name": "contract.pdf",
      "mimeType": "application/pdf",
      "size": 248901,
      "iv": "Vh3Q2M8Kz1pL0bR7"
    }
  ]
}
```

### Что происходит

1. Генерируется AES-ключ сообщения.
2. Payload шифруется в `encryptedPayload`.
3. Сам PDF читается как `ArrayBuffer`.
4. PDF шифруется через `AES-GCM` с `attachments[0].iv`.
5. На сервер загружается уже зашифрованный бинарный `.bin` файл.

### Что оказывается на сервере

В таблице `messages`:

```json
{
  "kind": "file",
  "text": "",
  "fileUrl": "/uploads/1746723456-1.bin",
  "fileName": "",
  "encryptedPayload": "WmF6c2xQbG9hZC1jaXBoZXJ0ZXh0",
  "encryptionMeta": "{\"version\":1,\"algorithm\":\"AES-GCM+RSA-OAEP-256\",\"iv\":\"3bL4YwR9zQp1nHmX\",\"keys\":{\"1\":\"...\",\"2\":\"...\"}}"
}
```

А реальный файл на диске уже выглядит как бессмысленный набор байт:

```text
8F 21 C3 59 A0 7D 4E 10 8B 4A 92 ...
```

Сервер не знает:

- что файл назывался `contract.pdf`;
- что это был PDF;
- что в подписи было `"Подписанный договор"`.

Он знает только:

- что пришло вложение типа `file`;
- где лежит зашифрованный blob;
- кто и куда его отправил.

---

## Поток чтения сообщения

Когда получатель открывает чат:

1. клиент получает `encryptedPayload` и `encryptionMeta`;
2. из `encryptionMeta.keys[currentUserId]` берётся AES-ключ, зашифрованный именно для этого пользователя;
3. этот AES-ключ расшифровывается локальным `privateKey`;
4. затем этим AES-ключом расшифровывается `encryptedPayload`;
5. если есть вложения, клиент скачивает зашифрованный файл и расшифровывает его локально;
6. для показа создаётся `Blob` и временный `objectURL`.

Схема:

```text
server -> encryptedPayload + encryptionMeta + fileUrl
client -> decrypt wrapped AES key using privateKey
client -> decrypt payload using AES key
client -> decrypt file bytes using AES key + attachment IV
client -> render plaintext UI
```

## Наглядный пример: одно и то же сообщение на разных этапах

### Что вводит пользователь

```text
"Встреча в 19:30, код подтверждения 4819"
```

### Открытый payload до шифрования

```json
{
  "text": "Встреча в 19:30, код подтверждения 4819",
  "attachments": []
}
```

### После шифрования

```json
{
  "text": "",
  "encryptedPayload": "Aq9C2x8B1wGmP0yA8n6wY6TnR8r0D4s7f0B...",
  "encryptionMeta": "{\"version\":1,\"algorithm\":\"AES-GCM+RSA-OAEP-256\",\"iv\":\"j5X0mQ2aL7wN3bR1\",\"keys\":{\"1\":\"fJ3...\",\"2\":\"sK9...\"}}"
}
```

### Что видит сервер

```text
chatId = 15
senderId = 2
kind = text
text = ""
encryptedPayload = "Aq9C2x8B1wGmP0yA8n6wY6TnR8r0D4s7f0B..."
encryptionMeta = "{...}"
```

### Что видит получатель после расшифровки

```text
Встреча в 19:30, код подтверждения 4819
```

## Как работает редактирование

При редактировании текст не меняется на сервере в открытом виде.

Вместо этого:

1. пользователь вводит новый текст;
2. фронтенд снова формирует payload;
3. создаётся новый AES-ключ;
4. новый payload заново шифруется;
5. сервер сохраняет новый `encryptedPayload` и `encryptionMeta`.

То есть редактирование фактически создаёт новую зашифрованную версию содержимого.

---

## Как работает удаление

При удалении сервер очищает:

- `text`;
- `file_url`;
- `file_name`;
- `encrypted_payload`;
- `encryption_meta`;

и выставляет `deleted_at`.

После этого расшифровывать уже нечего: контент считается уничтоженным на серверной стороне.

---

## Что происходит, если у участника нет ключа

Если у любого участника чата нет `publicKey`, клиент блокирует отправку и показывает причину.

Почему это важно:

- нельзя безопасно зашифровать сообщение для пользователя без его публичного ключа;
- частичное шифрование для части участников ломало бы модель E2EE.

Поэтому текущая логика строгая:

- либо все участники готовы к E2EE;
- либо сообщение не отправляется.

---

## Что происходит, если пользователь зашёл с другого устройства

Сейчас приватный ключ хранится локально в браузере.

Это значит:

- новый браузер или новое устройство не имеют старого `privateKey`;
- старые E2EE-сообщения на таком устройстве не расшифруются;
- интерфейс покажет, что сообщение зашифровано, но расшифровка недоступна.

Это ограничение текущей версии.

Для production-уровня обычно добавляют:

- экспорт/импорт ключа;
- резервное шифрованное хранилище ключей;
- подтверждение новых устройств;
- отдельные ключи по устройствам, а не по пользователю.

---

## Что видно злоумышленнику при компрометации сервера

Если кто-то получит доступ только к базе и файловому хранилищу, он увидит:

- список пользователей;
- публичные ключи;
- список чатов и участников;
- ciphertext сообщений;
- ciphertext вложений;
- время отправки и размеры.

Он не увидит:

- текст сообщений;
- реальные имена и типы вложений;
- содержимое файлов;
- подписи к изображениям и файлам;
- длительность и содержимое голосового как понятный контент без расшифровки payload.

## Короткая схема всей системы

```text
1. Пользователь логинится
2. Клиент проверяет локальную пару ключей
3. Если ключей нет:
   - генерирует RSA key pair
   - сохраняет private/public locally
   - отправляет publicKey на сервер

4. При отправке сообщения:
   - генерируется AES key
   - payload шифруется AES-GCM
   - AES key шифруется publicKey каждого участника
   - сервер получает только ciphertext

5. При чтении:
   - клиент берёт свой wrapped AES key
   - расшифровывает его privateKey
   - расшифровывает payload и вложения
   - показывает plaintext только локально
```

## Важное замечание

Это уже настоящая клиентская E2EE-модель для содержимого сообщений и файлов, но пока без полноценных device-management возможностей.

Текущая реализация хороша для демонстрации и MVP, но для production обычно ещё нужны:

- ротация ключей;
- поддержка нескольких устройств на пользователя;
- верификация ключей собеседника;
- защита от подмены публичного ключа сервером;
- безопасный экспорт/бэкап приватных ключей;
- forward secrecy и более продвинутый протокол вроде Double Ratchet.

Если хочешь, следующим сообщением я могу ещё сделать вторую версию этого файла:

- в виде архитектурной схемы;
- или в виде документа для не-технических пользователей;
- или в виде документации уровня “для разработчика проекта”.
